About the AI Vibe Code Repository

A library-curated home for interactive web applications created through human expertise, conversational AI coding, iterative testing, open documentation, and responsible sharing.

What Is the AI Vibe Code Repository?

The Lewis University Library AI Vibe Code Repository is a curated showcase of interactive web applications built by Lewis University faculty, students, librarians, and staff using generative AI tools such as ChatGPT, Claude, Cursor, Replit, Gemini, Copilot, and related platforms. It preserves not only finished apps, but also the reasoning behind them: the learning problem, the human expertise, the AI-assisted development process, and the metadata that makes each project reusable.

Rather than treating AI as a content generator alone, the repository frames AI as a development partner that helps community members turn ideas into functional, browser-based tools for teaching, learning, research, and university operations.

The core shift: from technology adopters to technology creators. Lewis community members do not have to wait for a commercial product to solve a local teaching or workflow problem; they can prototype, test, document, and share a focused tool built around their own context.

Research Grounding

This redesign draws on three overlapping arguments in the vibe-coding literature. Pesce and Cheungpasitporn (2025) describe vibe coding as a clinician-led, AI-assisted method for converting static expert knowledge into interactive, open-source web tools that mirror expert reasoning and support transparent digital scholarship. Royal (2026) connects vibe coding to flow theory, emphasizing project-based learning, immediate feedback, challenge-skill balance, creativity, and student agency in AI-augmented coding education. Chow and Ng (2025) frame AI-assisted vibe coding as a way for educators without formal programming backgrounds to create customized simulations through structured prompts, rapid prototyping, and iterative refinement.

Together, these sources support a repository model that values domain expertise, learner experience, transparent process, and reusable source code as much as the application itself.

  • Pesce & Cheungpasitporn (2025). Vibe Coding in nephrology education: clinician-led, AI-assisted development of open-source interactive learning tools. Used here to frame expert reasoning, open-source deployment, and transparent digital scholarship.
  • Royal (2026). Integrating Vibe Coding and Flow Theory: A Student-Centered Model for AI-Augmented Coding Education. Used here to frame project-based learning, flow, feedback, agency, and AI-supported coding confidence.
  • Chow & Ng (2025). From technology adopters to creators: Leveraging AI-assisted vibe coding to transform clinical teaching and learning. Used here to frame educator-led prototyping, accessibility for non-programmers, and the move from adopting tools to creating them.
  • Lewis University Library (n.d.). Submission Guide & Governance: Lewis University Vibe Code Showcase. Used here to shape local eligibility, metadata, review, hosting, and submission workflow logic.

What Belongs Here?

The repository is intentionally focused: it showcases interactive web applications, not every possible AI-generated artifact. Strong submissions are functional in a browser, built substantially with AI assistance, grounded in a pedagogical or institutional purpose, appropriate for public display, and documented well enough that another person can understand the human-AI collaboration behind the tool.

Who Can Contribute?

The repository is open to Lewis University faculty, students, librarians, staff, and administrators. Students may contribute with faculty endorsement. External collaborators can be credited, but each project needs a Lewis-affiliated primary contributor.

Guided Submission

Best for most contributors: submit app files or a live URL, describe the purpose and AI workflow, and the Library team reviews, catalogs, and publishes the project.

GitHub Workflow

Best for technical contributors: use the project template, include metadata and a README, and submit the project for review and repository inclusion.

Library Curation & Governance

Lewis University Library maintains the repository as a discoverability, preservation, and faculty-development infrastructure. The Library manages metadata consistency, controlled vocabularies, GitHub Pages publication, project descriptions, and review workflows. Review is lightweight and practical: the team checks whether the app loads, whether its purpose is clear, whether metadata and AI attribution are present, whether the content is appropriate for public display, and whether privacy or safety concerns need attention.

The Library does not replace the contributor’s domain expertise. Pedagogical effectiveness, factual accuracy, and disciplinary interpretation remain the responsibility of the creator or endorsing faculty member.

Technical Model

Projects are expected to run as static or lightweight web applications wherever possible: a single index.html, HTML/CSS/JS files, or a static build from frameworks such as React, Vue, or Svelte. GitHub Pages compatibility is the default assumption, which keeps hosting simple, sustainable, and open.

Each project should include clear metadata: title, description, author, department, discipline, AI tool used, content type, tags, live URL, source URL, license, and publication status. Screenshots and README documentation are strongly encouraged.

Contact & Questions

Have a vibe-coded app idea, a finished tool, or a question about what counts? Reach out through LibChat or visit the Lewis University Library website.

Repository Logic

How vibe-coded ideas become shareable tools

The repository is built around a simple idea: Lewis community members can turn local expertise, teaching needs, research questions, and workflow problems into browser-based tools through conversational AI-assisted development.

1. Start with a real use caseBegin with a concrete teaching, learning, research, library, or operational need.
2. Translate expertise into logicBreak domain knowledge into choices, pathways, prompts, variables, calculations, feedback, or user actions.
3. Build conversationallyUse tools like Claude, ChatGPT, Cursor, Replit, or Copilot to generate and revise code through natural-language instructions.
4. Test the vibeRefine the interface, accessibility, accuracy, tone, and user experience until the tool actually serves its audience.
5. Document the reasoningRecord the purpose, AI tool used, human revisions, target audience, and pedagogical or institutional context.
6. Share the sourcePublish the app, preserve the source, and make the build process discoverable so others can adapt it.

This turns the repository into more than a gallery: it becomes a reusable map of how expertise becomes interactive, open, and remixable.

Submission Guide

Lewis University Vibe Code Showcase: Submission Guide & Governance

Use this guide to review eligibility, prepare submission materials, understand review criteria, and reference governance policies for the Lewis University Vibe Code Showcase.

What this is: a curated Lewis University Library showcase of interactive web applications built by Lewis community members using generative AI tools. The purpose is faculty development, pedagogical innovation, and institutional knowledge-sharing.

Who can submit?

  • Faculty: course, research, or departmental tools.
  • Students: coursework, independent study, or personal learning projects with faculty endorsement.
  • Librarians: instruction, reference, discovery, or resource-management tools.
  • Staff: operational, student-service, or administrative workflow applications.

What can be submitted?

  1. Substantially built with AI assistance.
  2. Runs in a web browser as a static or lightweight web app.
  3. Serves a pedagogical, research, library, or institutional purpose.
  4. Is appropriate for public display.
  5. Loads, renders, and performs its core function.

What is not eligible?

  • Purely decorative or portfolio-only static pages.
  • Apps requiring accounts, payment, or restricted APIs to function.
  • AI text logs without functional application behavior.
  • Duplicate or near-duplicate submissions.
Submission workflow

Path A: Non-technical contributors

  1. Build your application using an AI tool such as Claude, ChatGPT, Cursor, Replit, Copilot, or another assistant.
  2. Export your files, including at minimum an index.html file, and gather HTML, CSS, JS, images, and assets into one folder.
  3. Submit your name, department, email, title, description, AI tool used, ZIP file or deployment link, discipline, content type, Bloom’s level, keywords, and optional screenshot.
  4. The library team reviews the submission.
  5. Upon approval, the library team creates or publishes the repository, enables the live site, and adds it to the gallery.

Path B: Technical contributors

  1. Fork or use the project-template repository.
  2. Add application files and complete project.json metadata.
  3. Complete the README template.
  4. Submit a pull request or repository for review.
  5. After approval, the project is published and added to the showcase.

Controlled vocabularies

Disciplines: Nursing; Natural Sciences; Humanities; Social Sciences; Aviation & Transportation; Mathematics & Statistics; Computer Science & IT; Business; Education; Criminal Justice; Interdisciplinary.

Content types: Simulation; Visualization; Discussion Tool; Reference Tool; Interactive Lab; Decision Tool; Creation Tool; Analysis Tool; Quiz; Calculator; Game; Other.

Bloom’s levels: Remember; Understand; Apply; Analyze; Evaluate; Create.

Review process

The library review is a lightweight quality check, not a peer review of pedagogical effectiveness.

CriterionWhat is checked
FunctionalityDoes the app load and perform its core function?
PurposeIs there a clear pedagogical or institutional reason for the app?
MetadataAre description, discipline, tags, type, and status appropriate?
ContentIs the content appropriate for public university display?
AI attributionIs the AI tool identified?

Timeline: initial review within 5 business days; publication within 2 business days of acceptance.

Roles, policy, and technical notes

Library team

Manages the showcase site, reviews submissions, publishes approved projects, maintains metadata consistency, promotes the repository, and archives inactive or broken projects with contributor notification.

Contributors

Build and submit applications, provide accurate metadata, respond to review feedback, and optionally maintain projects over time.

Policy reminders

  • Contributors retain ownership while granting the library a non-exclusive license to host and display the work.
  • Applications must not include protected, confidential, FERPA, HIPAA, proprietary, or unauthorized copyrighted content.
  • Apps should not collect, store, or transmit personally identifiable information without clear disclosure.
  • GitHub Pages-compatible projects should use static HTML/CSS/JS, relative paths, and no server-side dependencies.
